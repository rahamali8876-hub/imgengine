# 1. RFC OVERVIEW

## 1.1 Problem

Traditional systems:

* ❌ malloc fragmentation → latency spikes
* ❌ scalar execution → low SIMD utilization
* ❌ unsafe parsing → security risks
* ❌ poor multi-tenant scalability

---

## 1.2 Objective

Build:

* ⚡ < 2ms latency (4K image)
* ⚡ 100K+ ops/sec
* ⚡ ZERO allocation in hot path
* ⚡ deterministic execution

---

## 1.3 Non-Goals

* GPU-first execution
* GUI tools
* general-purpose editing

## 6.2 CORE INNOVATION — FUSED PIPELINE

Instead of:

```
DRAM → CPU → DRAM → CPU → DRAM
```

We do:

```
DRAM → CPU (REGISTER) → DONE
```

✔ single-pass execution
✔ minimal memory bandwidth
✔ deterministic latency

---

## 6.3 PREPARED REGISTRY + PREPARED DECODE PATH

Runtime now owns a prepared template registry and a prepared decode ingress model.

HLD:

```text
CLI/API Request
  ↓
TemplateKey
  ↓
Runtime Template Registry
  ↓
Prepared Job Entry
  ↓
Prepared Decode State
  ↓
Pipeline / Arch Dispatch
```

RULES:

* presets are runtime-owned entries, not only inline job defaults
* API decode ingress may use prepared decoder state
* thread-local slab + thread-local arena back API decode/prepare state
* template lookup is constant-time over a fixed registry

RESULT:

* lower control-plane drift
* deterministic preset resolution
* cleaner separation of template identity vs job override values

---

## 6.4 RAW FRAME INGRESS

The engine now exposes a decode-bypass ingress for pre-decoded RGB24 frames.

HLD:

```text
CLI/API Request
  ↓
raw-rgb24 input contract
  ↓
input size + stride validation
  ↓
runtime render-stage prepare
  ↓
pipeline / arch dispatch
  ↓
encode or PDF handoff
```

RULES:

* raw ingress is explicit, never auto-detected
* width, height, and stride are validated before render-stage entry
* decoder is skipped entirely on this path
* raw ingress remains outside the hot path; pipeline and arch stay pure

RESULT:

* true decode-free CLI/API entry for prepared upstream frame producers
* cleaner separation of ingress cost vs render cost
* a direct path toward `< 2 ms` render-stage workloads when decode is not part of the contract

# 6.5 Pipelines and Registration

Pipeline registration must be explicit, C/assembly-first, and reviewable. Auto-generation via external scripts is allowed only when the resulting C is checked in or gated behind a maintainer-enabled option.

Guidance:
- Prefer hand-written `src/pipeline/*.c` registration functions that call `img_register_op()` and populate `img_job_t` templates using the canonical headers (`include/pipeline/job_presets.h`, `include/pipeline/job_templates.h`).
- If codegen is used, it must emit reviewed C (not runtime-parsed blobs) and the generated output should be committed to the repository or produced behind an opt-in CMake flag.
- The runtime must never parse DSL files at startup; any DSL-based workflow is a build-time convenience only.

Files of interest (canonical):
- `src/pipeline/*` — pipeline implementations and registration points
- `include/pipeline/job_presets.h` — canonical preset names and lookup helpers
- `include/pipeline/job_templates.h` — compatibility shim to presets
- `include/pipeline/jump_table.h` and `src/pipeline/jump_table_register.c` — hot-path jump table and registration

Policy:
- No Python or external codegen runtime dependencies allowed in production build containers unless those tools are part of the project maintenance toolchain and gated by CMake options.
- Generated C must follow the project's coding and review standards and be visible in PR diffs.

Rationale:
- Keep the hot path pure C/assembly and avoid runtime code parsing.
- Maintain reviewability and ABI stability.

## 7.8 L18 — LEAF ASSEMBLY KERNELS


PLACEMENT:

* row-copy leaf in `src/arch/x86_64/avx2/blit_avx2.c`

RULES:

* bounds and clipping remain in C
* assembly owns no policy
* no allocation
* no syscalls
* sanitizer builds fall back to C

RESULT:

* kernel-grade low-level optimization without sacrificing safety
* assembly remains a leaf implementation detail, not a control-plane dependency

# 8. SIMD ARCHITECTURE

---

## 8.1 SoA (STRUCTURE OF ARRAYS) — CRITICAL UPGRADE

### ❌ AoS (BAD)

```c
struct Pixel { uint8_t r,g,b; };
```

### ✅ SoA (CORRECT)

```c
struct ImageSoA {
    uint8_t* r;
    uint8_t* g;
    uint8_t* b;
};
```

✔ full SIMD width
✔ no shuffles
✔ perfect cache usage

---

## 8.2 AVX2 FUSED PIPELINE

```c
for (i += 32) {
    __m256i px = _mm256_load_si256(...);

    // fused ops:
    // crop + border + padding + bleed

    _mm256_store_si256(...);
}
```

---

## 8.3 AVX-512 FUSED KERNELS

```c
for (i += 64) {
    __m512i r = _mm512_load_si512(r_ptr);
    __m512i g = _mm512_load_si512(g_ptr);
    __m512i b = _mm512_load_si512(b_ptr);

    // fused operations

    _mm512_store_si512(r_ptr, r);
}
```

✔ 2× width vs AVX2
✔ mask registers → branchless
✔ fewer iterations

---

## 8.4 AVX-512 BILINEAR RESIZE (PRODUCTION PATH)

The AVX-512 resize path is now a real 16-lane bilinear kernel shape instead of a placeholder stub.

```c
for (dx += 16) {
    // 16-lane weight setup
    // 16-lane top/bottom blend
    // 16-lane final vertical blend
}
```

CURRENT CHARACTERISTICS:

* 16-lane integer blend path
* uses precomputed x/y index and weight tables when available
* scalar tail for remainder pixels
* same kernel ABI as AVX2/scalar

RESULT:

* AVX-512 is now on the real execution path
* wider lanes are used for resize math
* jump table can route resize to a production AVX-512 implementation

## SOLID PRINCIPLES (APPLIED TO C & ASSEMBLY)

The classical SOLID principles are adapted here for a low-level, kernel-grade C and assembly codebase where performance, determinism, and minimal runtime overhead are paramount.

* **S — Single Responsibility:** each module or translation unit (`.c` / `.h`) owns one responsibility (e.g., `slab.c` — memory slab management; `scheduler.c` — NUMA scheduling). Keep assembly files limited to validated leaf kernels only.
* **O — Open/Closed:** prefer table-driven, jump-table extensibility and codegen (DSL → generated pipelines) so new pipelines or kernels are added without modifying hot-path logic; keep interfaces stable and versioned. ( job_job_presets.h / job_job_templates.h) here codebases present for this.
* **L — Liskov Substitution (Practical):** design kernel interfaces so fallback scalar implementations can substitute SIMD/AVX implementations without changing caller behavior; ensure ABI stability for generated pipeline functions.
* **I — Interface Segregation:** expose minimal C APIs across the API/runtime/pipeline boundary. Keep hot-path interfaces lean (plain POD structs, function pointers, no heap ownership) and move richer functionality to control plane code.
* **D — Dependency Inversion:** depend on abstractions (function pointers, jump tables, small vtables for plugins) rather than concrete implementations; load concrete NUMA, arch, or kernel implementations at startup via registry so runtime and pipelines remain decoupled.

Notes:

* Keep SOLID guidance pragmatic: favor explicit, low-overhead patterns compatible with zero-allocation hot paths.
* Use codegen to enforce interface stability and to generate safe fallbacks for sanitizer/portable builds.
